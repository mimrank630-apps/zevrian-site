# Static e-commerce site for Zevrian Global LLC

A complete greenfield static website for a premium lifestyle brand, built as 10 HTML pages, a CSS design system, and 4 JavaScript modules with no build step. The site uses directory-based routing for Cloudflare Pages, renders product grids dynamically from a JS data file, and includes a client-side checkout modal that collects raw credit card numbers without any payment processor integration. The approach favors simplicity and fast deployment at the cost of payment security and SEO discoverability.

**Watch for:** The checkout modal collects card details directly into DOM inputs with no payment processor (Stripe, Braintree, etc.), making the "Buy Direct" flow a PCI compliance hazard if deployed as-is (confirmed). The "Buy on Amazon" buttons link nowhere — there are no actual Amazon URLs, only a placeholder string in the product detail and a toast message on listing pages (confirmed). Product detail pages are fully client-rendered from query params, meaning search engines and social crawlers see an empty container (confirmed).

## High-level view

The product data architecture stores all 18 products in a single JS array loaded globally, then uses data attributes on empty grid containers to signal which products should render on each page. Adding or modifying products requires editing source code — there's no CMS layer or JSON feed.

Payment handling is the primary risk. The checkout modal renders Stripe-style card inputs that collect cardholder name, card number, expiry, and CVV directly into the DOM, then "processes" the order with a setTimeout. No data is transmitted anywhere, so the checkout is purely cosmetic — but if any backend integration is added later without replacing this form with tokenized hosted fields, the site will be handling raw card data in violation of PCI-DSS requirements.

The site's SEO posture is split: static pages (homepage, categories, legal) have proper meta tags and server-rendered content. But the product detail page serves an empty HTML shell and relies on JavaScript to populate content, title, and meta tags at runtime. Crawlers that don't execute JS will see a blank page.

Forms collect data but discard it. Both contact forms and the newsletter validate client-side and show a toast, but no data is stored, emailed, or submitted to any endpoint.

1. **Raw card collection without payment processor** — Replace the checkout form's native `<input>` fields with Stripe Elements or equivalent hosted fields so card data never touches the merchant's DOM. Until then, disable or remove the "Buy Direct" flow.
2. **Amazon buttons link nowhere** — Product detail "Buy on Amazon" links use `href="https://amazon.com/dp/PLACEHOLDER"`. Listing page Amazon buttons only fire a toast. Provide actual affiliate/product URLs or remove the buttons.
3. **Product detail pages invisible to crawlers** — The `/shop/product/index.html` shell has no server-rendered product content. Implement pre-rendering, static generation per product, or at minimum add structured `<noscript>` fallback content.
4. **Forms submit to nowhere** — Contact forms and newsletter call `preventDefault()` and show a toast but never POST data. Wire up a form handler (Cloudflare Workers, Formspree, etc.) or document that these are placeholders.
5. **No HTTPS/CSP/security headers configured** — No `_headers` file or equivalent for Cloudflare Pages. Add Content-Security-Policy, X-Frame-Options, and other hardening headers before launch.
6. **No favicon or Open Graph images** — No `<link rel="icon">` on any page; no `og:image` meta tags. Social shares and browser tabs will show generic defaults.
7. **PayPal button is non-functional** — The PayPal checkout button shows a "coming soon" toast. Either integrate PayPal SDK or remove the button to avoid misleading users.

## Checkout modal and PCI exposure

The checkout flow in `cart.js` constructs a modal with standard `<input>` elements for card number, expiry, and CVV. Validation checks field lengths (13-16 digits for card number, 4 digits for expiry, 3-4 for CVV) and that the expiry month is between 01-12.

On "successful" validation, the form handler disables the submit button, waits 1500ms, then shows a success toast and closes the modal. No data leaves the browser. The card values sit in DOM inputs that are never cleared beyond a `form.reset()` after the next modal open. If this flow were ever wired to an API endpoint, raw card data would be transmitted from the merchant's page — a PCI-DSS violation that shifts the merchant from SAQ A (hosted fields) to the full SAQ D assessment.

The PayPal button beside the card form fires `showToast('PayPal checkout coming soon.', 'info')` — a stub that will confuse real users expecting to complete a purchase.

## Product detail rendering and SEO gap

The product detail page (`/shop/product/index.html`) serves a shell:

```html
<main class="product-detail-page">
  <div class="container" id="product-detail-container">
    <!-- Content rendered by product-detail.js -->
  </div>
</main>
```

`product-detail.js` reads the `?id=` query parameter, looks up the product in the global `zevProducts` array, then injects the full product page. It also updates `document.title` and the `og:title`/`og:description` meta tags dynamically.

Consequences:
- Googlebot's initial HTML fetch sees no product content, no title, no structured data.
- Social share previews (Facebook, Twitter, Slack) that don't execute JS will show "Product - Zevrian" with a generic description.
- Users with JS disabled see an empty page with no `<noscript>` fallback.

For a static site targeting Cloudflare Pages, the fix is either pre-rendering a separate HTML file per product during a build step, or using Cloudflare Workers to server-render the page on request.

## Amazon integration gap

On product listing pages, the "Buy on Amazon" button handler in `main.js` responds to clicks on `.btn-amazon` by showing `showToast('Redirecting to Amazon...', 'info')` — but never navigates anywhere.

On the product detail page, the Amazon button is a real `<a>` tag but points to a placeholder:

```javascript
html += '<a href="https://amazon.com/dp/PLACEHOLDER" class="btn btn-primary btn-lg btn-amazon-detail" ...>Buy on Amazon</a>';
```

The product data objects have no `amazonUrl` or `asin` field, so there's no mechanism to provide real links per-product. This is the site's primary purchase CTA and it goes nowhere.

## Form data loss

Both contact forms and the newsletter validate inputs and show success toasts, but `e.preventDefault()` stops default form submission and no XHR/fetch call replaces it. Submitted data evaporates.

For Cloudflare Pages deployment, these forms need either a Cloudflare Workers function endpoint, a third-party service (Formspree, etc.), or at minimum a `mailto:` action fallback.

## Deployment configuration gaps

Missing deployment artifacts:
- No `_headers` file for security headers (CSP, HSTS, X-Content-Type-Options, X-Frame-Options, Referrer-Policy). The checkout modal especially needs a restrictive CSP since it handles form inputs that look like payment fields.
- No `robots.txt` or `sitemap.xml` for search engine guidance.
- No favicon (`<link rel="icon">`), which every browser tab and bookmark will show as missing.
- No `og:image` meta tags on any page, so social shares will have no visual preview.

## Header/footer template duplication

Every HTML page contains the identical header (logo SVG, desktop nav, mobile nav, hamburger) and footer. Any navigation change (adding a page, renaming a category) requires editing all 10 HTML files in lockstep — a maintenance cost inherent to the "no build step" constraint that grows with each new page.
